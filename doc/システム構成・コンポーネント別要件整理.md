# システム構成・コンポーネント別要件整理

## 全体アーキテクチャ概要

本システムは Envoy を API Gateway としたマイクロサービス構成であり、  
Frontend / API Gateway / app API / Auth Service / AI Service / Redis / Observability により構成される。

API Gateway に認証・認可・レート制限・WebSocket Upgrade を集約し、  
各サービスは疎結合で通信する。

---

## 1. Frontend（SPA / React）

### 役割

- チャット UI の提供
- REST API / WebSocket のクライアント
- session token の保持・送信

### 機能要件

- OpenAPI 定義から生成された API Client を使用
- REST API 呼び出し
- WebSocket によるリアルタイム通信
- 認証エラー（401 / 403）のハンドリング
- レート制限（429）のハンドリング
- 5 人マッチング完了時の自動画面遷移

### 使用技術

- React
- TypeScript
- Vite
- Tailwind CSS
- openapi-generator（typescript-fetch）
- WebSocket API（Browser）

### 通信の流れ

1. CloudFront 経由で SPA を取得
2. S3 から OpenAPI 定義（api.yml）を取得
3. Envoy Gateway に REST / WebSocket リクエスト送信
4. Authorization ヘッダーに session token を付与

### 使用プロトコル

- HTTPS（REST）
- WebSocket（ws / wss）
- HTTP/2（Envoy 経由）

---

## 2. Envoy API Gateway

### 役割

- 外部トラフィックの単一入口
- 認証・認可・レート制限・プロトコル制御
- 内部マイクロサービスへのルーティング

### 機能要件

- JWT 検証（jwt_authn）
- HTTP → WebSocket Upgrade
- REST / gRPC ルーティング
- レート制限（Local Rate Limit）
- Metrics / Tracing の出力

### 使用技術

- Envoy Proxy
- Kubernetes
- jwt_authn filter
- local_rate_limit filter
- OpenTelemetry

### 通信の流れ

1. Frontend からの HTTP / WebSocket を受信
2. jwt_authn で token 検証
3. レート制限チェック
4. app / AI Service に転送
5. Metrics / Trace を Observability に送信

### 使用プロトコル

- HTTPS / HTTP/2
- WebSocket
- gRPC（内部通信）
- OTLP（Tracing / Metrics）

---

## 3. app API Service（Go）

### 役割

- ビジネスロジックの中核
- マッチング・ルーム管理・メッセージ制御

### 機能要件

- OpenAPI 準拠の REST API
- WebSocket サーバー
- 5 人ランダムマッチング
- Redis Pub/Sub / Streams 連携
- AI Service との連携

### 使用技術

- Go
- net/http / chi / gin
- WebSocket（gorilla/websocket / nhooyr）
- Redis Client（go-redis）
- OpenAPI Generator
- OpenTelemetry SDK

### 通信の流れ

1. Envoy から REST / WebSocket リクエスト受信
2. 認証済みリクエストのみ処理
3. マッチング状態を Redis に書き込み
4. メッセージを Redis Streams に push
5. WebSocket 経由でクライアントに配信

### 使用プロトコル

- HTTP/JSON
- WebSocket
- gRPC（AI Service 連携）
- Redis Protocol
- OTLP

---

## 4. Auth Service（Go）

### 役割

- ログイン / サインアップ API 提供
- JWT 発行
- リバースプロキシの認証ゲートと連携

### 機能要件

- `/auth/login` / `/auth/signup` の REST API
- JWT 署名（HS256）
- JWT 期限管理（TTL）
- インメモリユーザー管理（将来永続化）
- ヘルスチェック `/healthz`

### 使用技術

- Go
- net/http
- JWT（golang-jwt/jwt）
- bcrypt（x/crypto）

### 通信の流れ

1. Frontend が `/auth/login` または `/auth/signup` を呼び出し
2. Auth Service がユーザー認証・登録を実施
3. JWT を発行して返却
4. 以降の API は Envoy が JWT を検証

### 使用プロトコル

- HTTP/JSON

---

## 5. AI Service（Python）

### 役割

- チャット内容解析
- レコメンド / モデレーション / RAG 処理

### 機能要件

- app からの同期 / 非同期呼び出し
- Redis データ参照
- 処理結果の返却 / publish
- スケーラブルな構成

### 使用技術

- Python
- FastAPI / gRPC
- Redis Client
- OpenTelemetry

### 通信の流れ

1. app から gRPC / HTTP 呼び出し
2. Redis から関連データ取得
3. AI 処理実行
4. 結果を Redis に publish または app に返却

### 使用プロトコル

- gRPC
- HTTP/JSON
- Redis Protocol
- OTLP

---

## 6. Redis（Cluster / Streams / PubSub）

### 役割

- 高速な状態管理
- 非同期イベント配信
- マッチング・メッセージングの中核

### 機能要件

- マッチングキュー管理
- ルーム / ユーザー状態管理
- メッセージ一時永続
- Broadcast イベント配信

### 使用技術

- Redis Cluster
- Redis Streams
- Redis Pub/Sub

### 通信の流れ

- app / AI Service から read/write
- Stream にイベント追加
- Pub/Sub で WebSocket broadcast

### 使用プロトコル

- Redis Protocol（TCP）

---

## 7. Observability（Prometheus / Grafana / Jaeger）

### 役割

- システム全体の可視化
- 障害解析・性能分析

### 機能要件

- メトリクス収集
- 分散トレーシング
- ダッシュボード表示
- アラート（将来対応）

### 使用技術

- Prometheus
- Grafana
- Jaeger
- OpenTelemetry

### 通信の流れ

- Envoy / app / AI Service が Metrics / Trace を送信
- Prometheus が scrape
- Grafana / Jaeger で可視化

### 使用プロトコル

- HTTP（metrics scrape）
- OTLP（gRPC / HTTP）

---

## まとめ

本システムは Envoy を API Gateway とすることで、  
認証・レート制限・WebSocket 制御を集約し、  
app / AI / Redis を疎結合に連携させることで  
高スケーラビリティかつリアルタイム性の高いチャットシステムを実現する。
